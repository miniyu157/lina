- [完成] update: 获取发行版数据（更新远程源索引）。
  遍历 repos 中的 URL，获取 INDEX 索引文件，验证 INDEX 版本标识。
  解析每一行数据（CSV格式：`<path>,<archs>,<versions>  # <description>`），
  得到 `PATH、NAME、ARCHS、VERSIONS、DESC`。
  对 `NAME、ARCHS、VERSIONS、DESC` 获取前八位 md5。
  存储发行版文件为 `${NAME}.${arch}.${content_hash}`。

目前解析INDEX索引的 CSV 后，得到
PATH、NAME、ARCHS、VERSIONS、DESC。对
NAME、ARCHS、VERSIONS、DESC 获取前八位
md5。然后又按照 arch 拆分文件。设计的是否不合
理？按照arch拆分有意义吗？
如果下游要使用--arch指定，或者自动根据uname -m搜索索引。那么这个update命令应该怎样拆分？

应该可以直接将INDEX索引集合。通过update命令编排到本地数据库文件。本地数据库文件被设计为每一行一个发行版：
source  path name(根据path计算)  archs  versions  desc  hash=md5的前八位(name+archs+versions+desc)
因为真正的下游数据，需要从url与path拼接，获取真正的发行版定义sh文件。
这个sh文件支持传入INDEX索引中的archs和versions，计算后得到原始的TAR_URL SUM_VAL SUM_CMD，这些不需要上游update关心，是需要下游pull/images等命令消费的数据。
下游pull/images需要搜索本地数据库，匹配发行版 name、archs、versions、desc。
这些数据是上游update需要编排的数据，所以用它们作为唯一hash；而path用于获取name、以及结合source获取下游sh定义文件）。
update 真正需要关心的是什么？

hash=md5的前八位(name+archs+versions+desc),source,path,name(根据path计算),archs,versions,desc
所以update的真正作用是将各个INDEX编排到一个本地大号INEDX。

原始INDEX为<path>,<archs>,<versions>  # <description>
应该把DESC作为数据，而不是注释，减少下游解析成本。
使用标准、严格的 tsv 数据存储。
那么现在需要重新定义数据结构：

INDEX(tsv) 每行:
<hash=md5(name+archs+versions+desc)>	<path>	<name>	<archs>	<versions>	<desc>

d/update 编排:
<hash(INDEX远程)>	<sh_url(source+path)>	<name>	<archs>	<versions>	<desc>

这里的 hash 相当于镜像ID，作为唯一标识符。

可以通过简单的拼接和编排，便捷插入本地数据库。
